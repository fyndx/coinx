## Project Overview

CoinX is an offline-first personal finance mobile app built with React Native/Expo. It tracks transactions (income/expenses), products with price history, and stores. Data lives in local SQLite and syncs bidirectionally with a Supabase backend. Authentication is optional — users can use the app anonymously and claim their data later.

## Commands

```bash
# Development
pnpm start           # Start Expo dev server
pnpm ios             # Run on iOS simulator
pnpm android         # Run on Android emulator

# Code quality
pnpm lint            # Run Biome linter (double quotes, recommended rules)
pnpm type-check      # TypeScript check

# Database
pnpm generate        # Generate Drizzle migrations from schema changes
```

## Architecture

### App Initialization Flow

`app/_layout.tsx` → `rootStore.actions.startServices()`:
1. Load fonts and run database migrations (`AppModel`)
2. Initialize Supabase auth session (`AuthModel`)
3. Initialize `SyncManager` (restores deviceId, starts AppState foreground listener)
4. On first launch: seed default categories, products, and stores
5. Trigger initial sync if authenticated

The app entry point (`app/index.tsx`) redirects to currency selection on first use, otherwise to the transactions tab.

`app/_layout.tsx` also runs `useDrizzleStudio(expoDb)` in `__DEV__` mode for database inspection.

### State Management (LegendState)

The app uses `@legendapp/state` with class-based observable models. All models live in `src/LegendState/` and are instantiated once in `RootStore` (`src/LegendState/index.ts`).

- **RootStore** — singleton exported as `rootStore`; holds all models
- **Model pattern** — `observable()` for state, class methods for actions, computed getters for derived views
- **Screen models** — named `*ScreenModel` (e.g., `TransactionsScreenModel`), hold UI-specific state
- **Component wiring** — screens/components wrap with `observer()` from `@legendapp/state/react`; models are passed as props with `$` suffix (e.g., `transactionsScreenModel$`)
- **Singleton exception** — `storeModel$` in `Store/Store.model.ts` is exported as a module-level singleton (not on RootStore) because it has no UI-specific state

Babel plugin `@legendapp/state/babel` is required (configured in `babel.config.js`) for fine-grained reactivity.

### Database (Drizzle + SQLite)

- **Schema**: `db/schema.ts` — 6 tables, all with UUID primary keys, `syncStatus`, `deletedAt`, and `localOwnerId` fields
- **Client**: `db/client.ts` — exports `db` (Drizzle) and `expoDb` (raw expo-sqlite handle)
- **Repos**: `src/database/*Repo.ts` — pure database queries wrapped in Effect.ts; never contain business logic
- **Migrations**: `drizzle/` artifacts generated by `pnpm generate`; a custom UUID migration in `db/migrations/uuid-migration.ts` runs first

Tables: `coinx_category`, `coinx_transaction`, `coinx_product`, `coinx_store`, `coinx_product_listing`, `coinx_product_listing_history`

### Sync Service (Effect.ts)

Located in `src/services/sync/`. The `SyncManager` class (`manager.ts`) orchestrates all sync operations and is exported as a singleton from `src/services/sync/index.ts`.

**Sync flow**: Ensure device → Push pending local records → Pull remote changes → Mark pushed records as `synced` (atomic)

- **Push**: collects all records with `syncStatus = 'pending'`, POSTs to `/api/sync/push`
- **Pull**: POSTs `lastSyncedAt` to `/api/sync/pull`, applies changes in dependency order (categories/stores/products first, then transactions/listings)
- **Debounce**: `scheduleSyncAfterChange()` waits 2 s after the last call before syncing
- **Device registration**: first sync registers device via `/api/auth/device`; web uses a session-scoped ID

Effect.ts tagged errors are in `errors.ts`; storage helpers (AsyncStorage) in `storage.ts`.

### Navigation (Expo Router)

File-based routing in `app/`:
- `app/(tabs)/` — main tabs: transactions, products, budgets, settings (insights tab is hidden via `href: null`)
- `app/(auth)/` — sign-in, sign-up
- Root-level modals: `add-transaction`, `add-product`, `add-product-listing`, `edit-product-listing`, `add-store`, `categories`, `stores`, `currency-select`
- `app/playground/` — dev-only screen, only registered in `__DEV__`

Auth guard in `app/_layout.tsx` (`useProtectedRoute`) redirects authenticated users away from auth screens. Auth is optional; unauthenticated users are not forced to sign in.

### Styling

Uses `uniwind` (TailwindCSS for React Native) + `tailwind-variants`/`class-variance-authority` for component variants. Tailwind 4 — no `tailwind.config.js`; configure via CSS in `global.css`. Theme tokens are provided by `heroui-native/styles` (imported in `global.css`). The `cn()` utility in `src/utils/cn.ts` merges classes.

Custom UI primitives live in `src/Components/ui/` (Button, Input, Text).

## Key Patterns

### Effect.ts for Database/API Operations

Repos return `Effect` values that are composed and run with `Effect.runPromise()`:
```typescript
const result = await Effect.runPromise(getTransactions({ startDate, endDate }));
```
Use `pipe()` to compose effects. Errors use `Data.TaggedError` classes (see `src/utils/error.ts`, `src/database/Stores/StoresErrors.ts`, `src/services/sync/errors.ts`).

### Soft Deletes

Never hard-delete records. Always set both fields and mark pending for sync:
```typescript
.set({ deletedAt: new Date().toISOString(), updatedAt: new Date().toISOString(), syncStatus: "pending" })
```
Filter out deleted records with `isNull(table.deletedAt)` in all queries.

### Sync After Local Mutations

Call after any local database write/delete:
```typescript
syncManager.scheduleSyncAfterChange();
```

### Anonymous-to-Auth Data Migration

All records have a `localOwnerId` field. When a user signs in or signs up, `claimAnonymousData(userId)` in `src/services/sync/migration.ts` sets `localOwnerId` and marks all unclaimed records as `pending` so they sync on the next push.

### Currency Handling

Selected currency is persisted to MMKV storage via `AppModel`. Prices are stored in the smallest currency subunit (e.g., cents) and converted via `@coinify/currency` on display.

### UUID Generation

Always use `generateUUID()` from `src/utils/uuid.ts` (wraps `expo-crypto`'s `randomUUID`) when creating new records.

`@/*` maps to the project root (e.g., `@/src/`, `@/db/`).
